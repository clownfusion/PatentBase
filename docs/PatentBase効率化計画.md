# PatentBase 効率化計画

## Context

要件定義.md に記載の特許業務（先行文献調査・出願前調査・明細書作成資料・任意特許確認・中間処理）は、現状 J-PlatPat の手動操作と手作業でのドキュメント作成に依存しており、繰り返し作業が多く時間コストが高い。本プロジェクトではこれらを段階的にツール化・自動化し、各担当者が独立して利用できる Web アプリとして構築する。複数名が各自の環境で独立利用するユースケースを想定。

### 技術スタック選定理由

| レイヤー | 技術 | 理由 |
|---|---|---|
| バックエンド | Python + FastAPI | AI ライブラリとの親和性が高く、非同期処理対応 |
| DB | SQLite | サーバー不要でローカル動作、ユーザーごとに独立した DB ファイル |
| フロントエンド | HTML/JS (Phase1-2) → React (Phase3 以降) | 段階的に複雑さを増やす |
| 特許データ取得 | Playwright (J-PlatPat スクレイピング) | J-PlatPat に公式 API がないため |
| PDF 読み込み | pdfplumber / PyMuPDF | 明細書 PDF のテキスト抽出 |
| AI 分析 | Claude API (claude-sonnet-4-6) + プロンプトキャッシュ | 長文の明細書処理にキャッシュで効率化 |
| Word/PPT 出力 | python-docx, python-pptx | 先行文献調査レポート・明細書作成資料 |
| Excel 出力 | openpyxl | クレームチャート・特許リスト |

### ユーザー分離方針
- 各ユーザーが自端末でローカル起動（`localhost`）
- `data/{username}/patents.db` でユーザーごとに SQLite を分離
- 認証不要（ローカル起動前提）

## フェーズ別実装計画

### Phase 1: 基盤構築 + 任意特許確認（4. の要件対応）
**優先理由**: 単一特許の確認・要約は他フェーズの土台でもあり、即座に業務価値が出る

**実装内容:**
1. プロジェクト構造・依存関係のセットアップ (FastAPI, SQLite, Playwright, Claude API)
2. 3種のデータ入力対応:
   - 特許番号 → J-PlatPat 自動取得
   - PDF 読み込み（公開特許・手元の明細書 PDF）
   - Word (.docx) 読み込み（特許事務所作成の明細書案）
3. Claude API による特許要約・ポイント整理・請求項の構造化（テキスト + 図面画像を合わせてマルチモーダル分析。図面を含めることで機械・電気系等の精度向上が見込める。単一特許対象のため図面取得のコストは無視できる）
4. 図解の多層対応:
   - 請求項の構造・関連性は Mermaid でブラウザ表示（構造把握用）
   - Draw.io XML エクスポートで後編集可能
   - 原文書の図面参照（補助用途）
   - ※ AI が明細書全文から**新たな直感的概念図を生成**する SVG 出力は Phase 1 完了後に追加（原図が難解な場合の代替として。原図流用ではなく明細書本文から再解釈して図化する位置付け）
5. 基本 Web UI（入力 → 結果表示）

**成果物:**
- 番号 / PDF / Word を入力 → 要点サマリー・請求項構造図を Web 画面に表示
- 分析結果の Word または Excel エクスポート（後から再確認・共有用）
- 図解は Draw.io XML でエクスポート可能（diagrams.net で開いて編集可能）
- 原文書の図面を参照（補助用途）
- Phase 1 完了後に必要性を確認して SVG 生成を追加（明細書本文から再解釈した直感的概念図として）
- 対応要件: **4. 任意特許の内容確認**

### Phase 2: 母集団管理 + 1次調査ワークフロー（1-1, 1-2, 2-2, 2-3 の要件対応）
**実装内容:**
1. 調査案件の管理（案件名・調査目的・ステータス）
2. 検索式の登録・管理
3. J-PlatPat から検索式による一括取得・母集団作成
4. 1 次調査 UI（特許リストを表示し「関連あり/なし/保留」を記録）
5. AI による 1 次スクリーニング支援（関連性の自動判定候補提示）
6. 特許リスト Excel 出力

**成果物:**
- 母集団の一括取得 → AI 支援の 1 次スクリーニング → Excel エクスポート
- 対応要件: **1-1, 1-2, 2-2, 2-3**

---

### Phase 3: 詳細分析 + クレームチャート（1-3, 2-4 の要件対応）
**実装内容:**
1. 1 次調査通過特許の詳細分析 UI（一括分析は図面なし。特定特許の深掘り時はユーザーが「図面を含めて分析」を選択可能）
2. 自社製品仕様の入力・管理
3. クレームチャート自動生成（Claude API で請求項 × 製品構成を対比）
4. クレームチャート Excel 出力
5. 2 次調査結果・差異のまとめ機能

**成果物:**
- クレームチャート自動生成 → Excel ダウンロード
- 対応要件: **1-3, 2-4**

---

### Phase 4: 明細書作成資料生成（3. の要件対応）
**実装内容:**
1. 発明概要入力フォーム（権利化ポイント・先行文献との差異・効果）
2. 図解テンプレート（構成要素・関連性の Mermaid 図）
3. Claude API による明細書作成資料ドラフト生成
4. Word / PPT 出力（特許事務所提出用・社内共有用）

**成果物:**
- 発明情報を入力 → 明細書作成依頼資料の Word / PPT を自動生成
- 対応要件: **3. 特許出願前の資料作成**

2. 拒絶理由の妥当性分析（Claude API）
3. 意見書・補正書のドラフト生成
4. Word 出力

**成果物:**
- 拒絶理由を入力 → 意見書・補正書ドラフトの Word 生成
- 対応要件: **5. 中間処理**

## 横断的な設計方針（保守性・拡張性）

- **サービス層の分離**: スクレイパー・AI・ドキュメント生成は独立したサービスクラスとし、差し替え可能にする（例：J-PlatPat が変更されてもスクレイパーだけ修正）
- **AI プロバイダー抽象化**: `ai_analyzer.py` は AI プロバイダーインターフェースとして設計し、Claude API が利用できない場合は明確なエラーメッセージを返す。将来他プロバイダー（OpenAI 等）への切り替えが必要になっても実装の差し替えのみで対応可能にする
- **プロンプト管理**: Claude へのプロンプトは `prompts/` ディレクトリに外出しし、コードを変えずに調整可能にする
- **プロンプトキャッシュ**: 明細書全文など長い入力には Claude API のプロンプトキャッシュを活用しコスト削減
- **API 設計**: REST API を明確に定義し、将来 Claude Desktop や他クライアントからも呼び出せる構造にする
- **DB マイグレーション**: Alembic でスキーマ変更を管理し、フェーズをまたいでも既存データを維持する
- **段階的フェッチ（Lazy Loading）**: J-PlatPat へのアクセスはスクリーニング段階に応じて最小限に絞る
  - 母集団作成時（全件）: メタデータ + 要約のみ取得・保存（3,000件で約15〜30 MB）
  - 1 次スクリーニング: 保存済みデータを利用（J-PlatPat へのアクセスなし）
  - 2 次スクリーニング: 1 次通過分（全体の 10〜20%）に対してのみ明細書全文を取得
  - 図面: 閲覧時にオンデマンド取得（ローカル保存しない）
  - → 全体的な保存容量を 30〜50 MB 程度に抑制しつつ、J-PlatPat への集中アクセスも回避

## 検証方法

| フェーズ | 確認方法 |
|---|---|
| Phase 1 | 実際の特許番号（例: 特開2020-123456）と手元PDFを入力し、サマリーと請求項構造が正しく出力されるか確認 |
| Phase 2 前 | Phase 1 完了後、実際に数十件の特許を取得してデータ量を実測し、段階的フェッチの容量見込みを確認・調整 |
| Phase 2 | 検索式を入力して母集団取得 → 1次スクリーニング → Excel 出力の一連フローを確認 |
| Phase 3 | クレームチャートの内容が実際の請求項と製品仕様に対応しているか確認 |
| Phase 4 | 生成された Word/PPT の構成・内容が特許事務所への提出に使えるレベルか確認 |
| Phase 5 | 実際の拒絶理由通知書を用いて意見書ドラフトの品質確認 |

## Claude Code Skills として切り出すべき機能

本プロジェクトで実装する機能の中で、特許業務以外にも汎用的に使えるものは Claude Code Skill として独立させると、他プロジェクトでも再利用できる。

### 推奨 Skill 化候補

| Skill 名 | 概要 | 汎用性 |
|---|---|---|
| `diagram` | 技術文書・構成説明・請求項などのテキストから図を自動生成（Mermaid + Draw.io XML 選択可） | **高い**（システム設計・ワークフロー・特許・業務フロー等あらゆる場面で使える） |
| `summarize-doc` | 長文の技術文書（明細書・審査経過・仕様書等）を構造化サマリーに変換 | 高い（社内資料・報告書・マニュアルの読み込みにも使える） |
| `claim-chart` | 特許の請求項と製品仕様を対比したクレームチャートを生成 | 中程度（IP・法務領域に特化するが、契約条項比較など応用可能） |

## 計画の見直しについて

各フェーズ完了後に実測値・利用状況・新たな要件をもとに計画を適宜修正する。特に以下のタイミングで見直しを行う：
- Phase 1 完了後：実際のデータ量・AI 分析品質・J-PlatPat の挙動を確認し、Phase 2 の設計を調整
- 各フェーズ完了後：使い勝手・精度・追加要件を踏まえて次フェーズの仕様を見直す

## 着手優先度の根拠

Phase 1 から始める理由:
- 他フェーズ全てが依存する共通基盤（特許データ取得・Claude API・DB）を先に構築する
- 「4. 任意特許の内容確認」は独立した業務であり、基盤完成直後から実業務に使える
- フェーズをまたぐリファクタリングを最小化するため、データモデルと API 設計を最初に固める
